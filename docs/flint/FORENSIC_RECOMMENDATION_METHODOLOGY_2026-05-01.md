# Forensic Recommendation Methodology: Buy / Sell / Hold Logic

Date: 2026-05-01  
Repo: `/Users/sydneymilton/dev/_sandbox/tradingagents-flint-shadow`

## Scope

This document explains how the current TradingAgents shadow workflow produces
Buy / Overweight / Hold / Underweight / Sell recommendations, based on code and
observed run artifacts.

It distinguishes between:

- what the system actually computes or persists,
- what the LLM is instructed to infer,
- what is currently rigorous,
- what remains under-evidenced.

## Current Methodology Boundary

The method is not yet commercially robust. The strongest defensible claim is
that the workflow can produce a traceable advisory draft, subject to quality
gates and source provenance. It should not be treated as a validated trading
system until the final recommendation is tied to source IDs, target profile,
factor scoring, and replayable telemetry.

## End-to-End Decision Chain

The live service path is:

1. API receives a shadow-run request.
2. Worker claims the queued row.
3. `run_shadow_job(...)` builds a TradingAgents config.
4. `TradingAgentsGraph.propagate(ticker, trade_date)` executes the graph.
5. The graph writes a full-state JSON log and memory log.
6. The worker stores final rating, decision markdown, artifacts, and quality
   assessment.

The final rating is not calculated by a deterministic financial formula. It is
generated by the Portfolio Manager LLM, then parsed from the final markdown by a
deterministic rating parser.

## Agent Roles and Logic

### Market Analyst

Inputs:

- requested ticker,
- trade date,
- price/history data,
- technical indicators.

Tools:

- `get_stock_data`
- `get_indicators`

Purpose:

- summarize price trend, momentum, volatility, and volume-oriented signals.

Observed rigor:

- this stage can include concrete price/indicator tables.
- indicator selection is prompted, not fixed by a formal strategy.

Weakness:

- generated reports may include synthetic-looking code/table narrative rather
  than a clean trace from raw tool output to conclusion.

### News Analyst

Inputs:

- requested ticker,
- recent company-specific news,
- global/macro news.

Tools:

- `get_news`
- `get_global_news`

Purpose:

- summarize recent company and macro news relevant to the trade.

Observed rigor:

- can retrieve external news through configured vendor routes.

Weakness:

- the AAPL run showed severe entity contamination: Marvell and Avalyn content
  was treated as AAPL-relevant evidence. This is now guarded by prompt scope
  instructions and post-run quality checks, but source ranking remains weak.

### Bull and Bear Researchers

Inputs:

- market report,
- sentiment report,
- news report,
- fundamentals report,
- prior debate state.

Purpose:

- create adversarial arguments for and against investing.

Logic:

- LLM debate, not deterministic scoring.
- Bull is instructed to emphasize growth, competitive advantage, positive
  indicators, and rebut bears.
- Bear is instructed to emphasize risks, weak indicators, and rebut bulls.

Rigor:

- useful as structured adversarial reasoning.

Weakness:

- if earlier analyst reports are contaminated, bull/bear debate compounds the
  contamination.

### Research Manager

Inputs:

- bull/bear debate history.

Output schema:

- `recommendation`: Buy / Overweight / Hold / Underweight / Sell
- `rationale`
- `strategic_actions`

Logic:

- the LLM chooses a five-tier recommendation based on debate strength.
- `Hold` is reserved by prompt for genuinely balanced evidence.

Rigor:

- structured schema helps normalize output.

Weakness:

- if structured output fails, the system falls back to free text, and the run
  can still succeed.

### Trader

Inputs:

- Research Manager investment plan.

Output schema:

- action: Buy / Hold / Sell
- reasoning
- optional entry price
- optional stop loss
- optional position sizing

Logic:

- maps five-tier research view into a concrete transaction proposal.

Rigor:

- schema narrows transaction action to Buy/Hold/Sell.

Weakness:

- entry/stop/sizing are LLM generated, not validated against market data.

### Risk Analysts

Inputs:

- trader plan,
- analyst reports,
- debate history.

Roles:

- Aggressive: argues for upside/risk-taking.
- Conservative: argues for downside protection.
- Neutral: balances both.

Logic:

- LLM debate over risk posture.

Rigor:

- useful for surfacing competing risk frames.

Weakness:

- risk arguments are not yet scored or linked to hard risk metrics.

### Portfolio Manager

Inputs:

- Research Manager plan,
- Trader proposal,
- prior memory context,
- risk analyst debate.

Output schema:

- final rating: Buy / Overweight / Hold / Underweight / Sell
- executive summary
- investment thesis
- optional price target
- optional time horizon

Logic:

- the LLM synthesizes all prior outputs into the final decision.
- the service parses the final markdown to persist `final_rating`.

Rigor:

- this is the canonical final decision point.
- structured schema is useful when the provider follows it.

Weakness:

- fallback to free text can bypass schema guarantees.
- final claims are not yet linked to specific source objects or citations.
- price targets can be emitted without validation.

## Rating Semantics

The five-tier scale means:

- `Buy`: strong conviction to enter or add.
- `Overweight`: constructive view, increase exposure gradually.
- `Hold`: balanced view, maintain position.
- `Underweight`: cautious view, reduce exposure.
- `Sell`: strong conviction to exit or avoid.

Important: this is a prompt-level semantic scale, not a quantitatively calibrated
rating model.

## What Was Fixed

### 1. Ticker/entity scope guard

Added a reusable scope guard to keep every agent focused on the requested
instrument and reject out-of-scope sources.

Applied to:

- market analyst,
- news analyst,
- bull researcher,
- bear researcher,
- research manager,
- trader,
- aggressive risk analyst,
- conservative risk analyst,
- neutral risk analyst,
- portfolio manager.

### 2. Post-run quality assessment

Added deterministic quality checks for:

- unrelated ticker mentions,
- unrelated issuer mentions,
- missing intermediate reports,
- missing explicit source references.

The worker persists this under provider metadata as `quality`.

### 3. API/UI exposure

Decision and handoff endpoints now expose:

- `quality_status`
- `quality_findings`
- `source_summary`
- `recommendation_audit`

The UI decision panel now displays quality status, findings, and the
recommendation chain.

## Current Quality Status Semantics

- `passed`: no deterministic quality findings.
- `warning`: no hard contamination, but weaker evidence quality, such as no
  explicit source references.
- `failed`: hard defect, such as unrelated ticker/entity contamination or
  missing inspectable intermediate reports.

This separates pipeline completion from analysis trustworthiness.

## Recommendation Audit Semantics

The service now records a `recommendation_audit` object alongside quality
status. This object does not replace the model's recommendation. It explains
the decision chain that produced it:

- `research_manager_rating`: extracted from the Research Manager plan.
- `trader_action`: extracted from the Trader transaction proposal.
- `final_rating`: extracted from the Portfolio Manager decision.
- `directions`: normalized stance for each stage (`bullish`, `neutral`,
  `bearish`).
- `alignment_status`: `aligned` when research, trader, and final stages point
  in the same direction; `divergent` otherwise.
- `intermediate_report_count`: number of inspectable analyst reports available
  in final state.
- `explicit_source_reference`: whether the final decision contains explicit
  source-style language or URLs.

This makes a Buy / Sell / Hold recommendation easier to challenge. For example,
if the Research Manager says Buy, the Trader says Hold, and the Portfolio
Manager says Sell, the run can still complete but quality records
`recommendation_chain_divergent`.

## Critical Interpretation

The workflow has credible orchestration rigor:

- durable IDs,
- queued execution,
- event trail,
- state artifacts,
- hashed manifests,
- API handoff.

The recommendation methodology is still mostly qualitative LLM synthesis:

- no formal factor model,
- no deterministic scoring matrix,
- no calibrated probability estimate,
- no claim-level source citation,
- no strict source-to-recommendation trace.

Therefore the right interpretation is:

> The system can generate structured advisory hypotheses and preserve their
> evidence trail, but the Buy/Sell/Hold output must be treated as model-derived
> synthesis subject to quality gates and human review.

## Next Rigor Upgrades

1. Capture tool outputs as structured source objects.
2. Require final decisions to cite source object IDs.
3. Add deterministic ticker/entity validation before final synthesis.
4. Add a scoring layer for:
   - technical trend,
   - momentum,
   - volatility,
   - news sentiment,
   - fundamentals,
   - risk posture.
5. Require Portfolio Manager to reconcile its rating against the scoring layer.
6. Mark runs `quality_status=failed` when citations or ticker consistency fail.
