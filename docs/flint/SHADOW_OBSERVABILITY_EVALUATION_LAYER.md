# Shadow Observability And Evaluation Layer

Date: 2026-05-01  
Repository: `tradingagents-flint-shadow`

This repo now has a Tremor-style evaluation surface for TradingAgents shadow runs. The layer is deliberately advisory: it evaluates evidence quality, verification posture, scoring alignment, and judgement readiness. It does not place orders and does not write to Flint.

## What Was Ported From Tremor

The Tremor pattern is:

- rubrics define what good looks like
- evaluation runs bind a target artifact to a rubric and evaluator
- score rows preserve dimension-level judgement
- annotations create a queue/review signal such as `needs_revision` or `insufficient_evidence`
- datasets let selected runs become calibration examples
- phase-gate thinking keeps frontier-model signals subordinate to stored evidence and review outcomes

The Python/Postgres version here maps that onto shadow runs:

| Tremor concept | Shadow repo implementation |
| --- | --- |
| `evaluation_rubrics` | rubric records for shadow-run evidence and judgement criteria |
| `evaluation_runs` | one evaluation execution against a `shadow_run` target |
| `evaluation_scores` | dimension scores from 0-100 with confidence, basis, rationale, evidence JSON |
| `human_annotations` | review queue labels generated by system, LLM, or human evaluators |
| `evaluation_datasets` | calibration sets for accepted/rejected/ambiguous runs |
| `evaluation_queue_assignments` | queue assignment for failed or uncertain runs |

## API

Create an evaluation for a completed shadow run:

```bash
curl -sS -X POST "http://localhost:8000/v1/evaluations/shadow-runs/<run_id>" \
  -H "content-type: application/json" \
  -d '{"evaluator_type":"system","evaluator_model":"heuristic-v1"}'
```

List evaluations:

```bash
curl -sS "http://localhost:8000/v1/evaluations?shadow_run_id=<run_id>"
```

List rubrics:

```bash
curl -sS "http://localhost:8000/v1/evaluations/rubrics"
```

The operator UI at `/` also has an `Evaluation` panel. Load a run, then press `Run Evaluation`.

## Current Rubric

Default rubric: `shadow-run-frontier-readiness` version `2026-05-01`.

Dimensions:

- `evidence_traceability`: source-object and raw-tool citation discipline
- `ticker_scope`: requested ticker/entity consistency
- `scorecard_reconciliation`: Portfolio Manager rating versus deterministic scorecard
- `decision_readiness`: overall gate for Flint advisory ingestion and frontier-model review

Labels:

- `accept`
- `accept_with_notes`
- `needs_revision`
- `insufficient_evidence`
- `reject`

## Frontier-Model Contract

The storage contract is already frontier-ready. A frontier judge should read:

- run identity and selected analysts
- final decision markdown
- quality findings
- source summary
- recommendation audit
- artifact manifest

It should write:

- dimension scores
- overall label
- rationale
- cited evidence IDs
- unsupported-claim markers such as `unsupported_claim`, `insufficient_evidence`, or `should_retract_or_rewrite`

Until model-backed judgement is enabled, the implementation uses deterministic heuristic scoring and stores `basis="heuristic"`. Model-backed scores should use `basis="derived"` or `basis="authoritative"` only after calibration against stored datasets and human annotations.

## Methodology Requirements

Before a run can be treated as Flint-ready, the evaluation layer should also
check for:

- run-scoped telemetry and artifact hashes,
- explicit source IDs in the final recommendation,
- target-profile fields for the intended user,
- factor-model reconciliation against the final rating,
- explicit handling of stale or non-trading-day market data.

## Operating Boundary

Evaluation outputs are advisory evidence artifacts. Flint should ingest them only as judgement metadata unless Flint-side governance explicitly promotes a signal. Failed or insufficient-evidence labels should block use of the recommendation as a Flint advisory signal.
